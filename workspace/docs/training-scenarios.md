# 演習シナリオ集

> 対象: MCPツール活用研修  
> 前提: Open WebUI（http://localhost:3000）にアクセスできること

---

## 良いプロンプトの5要素

AIに質問する際、以下の5要素を意識すると回答の質が大きく向上します。

| 要素 | 書き方の例 | 役割 |
|------|----------|------|
| ① 役割 | 「あなたはSREエンジニアです」 | AIに専門家の視点を持たせる |
| ② 背景 | 「3月に障害が多発しており」 | AIが文脈を理解して回答できるようにする |
| ③ 依頼 | 「ログを分析してください」 | 何をしてほしいかを明確にする |
| ④ 制約 | 「ERRORレベルのみ対象、WARNは除く」 | 回答の範囲・条件を絞る |
| ⑤ 出力形式 | 「表形式で、件数・時間帯・原因の仮説を列挙」 | 使いやすい形式で受け取る |

**ポイント**: 5要素をすべて入れる必要はありません。  
「役割＋依頼＋出力形式」の3つを意識するだけで大幅に改善します。

## 演習1：ログ分析

### 目的
10日分のアプリケーションログを横断分析し、障害傾向を把握して報告書を生成する。

### 使用するMCPツール
- **filesystem**: `workspace/logs/` 内のログファイルを読み込む

### 悪いプロンプト例 ❌

```
ログ見て
```

**なぜ悪いか**: 何のログか、何を調べたいか、どんな形式で出力してほしいかが不明。  
AIは「何を見ればいいか」「何を答えればいいか」わからない状態になる。

---

### 良いプロンプト例 ✅

```
あなたはSREエンジニアです。

ECサイトの運用チームとして、3月1日〜10日のアプリケーションログ
（workspace/logs/app-2026-03-01.log 〜 app-2026-03-10.log）を全て読み込み、
以下の観点で障害分析を行ってください。

【分析項目】
1. ERROR・WARNの種別ごとの発生件数（日別）
2. 最も多く発生したエラーの種別と時間帯のパターン
3. 日をまたいで継続・悪化した障害の特定
4. 各障害の推定原因と影響範囲

【出力形式】
- 発生件数は表形式で
- 継続障害はタイムライン形式で
- 最後に「経営報告向け3行サマリー」を記載すること
```

**なぜ良いか**:
- ①役割：SREエンジニアとして専門的視点で分析させる
- ②背景：対象期間・ファイル名を明示
- ③依頼：4つの観点を具体的に列挙
- ④制約：分析の観点を構造化
- ⑤出力形式：表・タイムライン・サマリーと用途に合わせた形式を指定

---

### チャレンジ課題

ログ分析後、さらに以下を試してみましょう：

```
3/3〜3/5に集中した決済タイムアウト障害について、
発生時刻・件数・リトライ結果をまとめ、
「障害発生から復旧までの時系列」を記述してください。
```

## 演習2：DB操作

### 目的
SQLを書かずに自然言語でDBに問い合わせ、在庫確認・注文状況照会・売上集計を行う。

### 使用するMCPツール
- **sqlite**: `workspace/db/training.db` に対してSQLを自動生成・実行

### 悪いプロンプト例 ❌

```
DBどうなってる？
```

**なぜ悪いか**: 何を知りたいのか全く伝わらない。  
テーブル一覧を返してくれるかもしれないが、業務上役立つ情報は得られない。

---

### 良いプロンプト例 ✅（在庫確認）

```
あなたは在庫管理担当です。

productsテーブルとstock_historyテーブルを参照し、
以下の条件で商品リストを作成してください。

【条件】
- 現在の在庫数（stock_count）が20個以下の商品
- is_active = 1 の販売中商品のみ対象

【出力形式】
以下の列を含む表形式で、在庫数が少ない順に並べてください：
- 商品ID, SKU, 商品名, 在庫数, カテゴリ名
```

---

### 良いプロンプト例 ✅（注文状況照会）

```
あなたはカスタマーサポート担当です。

ordersテーブル・paymentsテーブル・shipmentsテーブルを結合し、
2026年3月1日以降に作成された注文について、
以下の情報を一覧表示してください。

【出力項目】
注文ID, ユーザーID, 注文日時, 注文ステータス, 決済ステータス, 決済方法, 配送ステータス

【特記事項】
- status が pending または confirmed の注文には「要対応」フラグを付けること
- 合計件数も最後に表示すること
```

---

### 良いプロンプト例 ✅（売上集計）

```
あなたは経営企画担当です。

orders・payments・order_items・productsテーブルを使い、
2026年3月の売上を以下の軸で集計してください。

1. 日別売上金額（tax込）と注文件数
2. カテゴリ別売上金額の合計（上位3位まで）
3. 決済方法別の利用件数と割合

結果は表形式で、最後に月間合計も表示してください。
```

---

### チャレンジ課題

```
最もレビュー評価が高い商品トップ3を、
平均評価・レビュー件数・現在の在庫数とともに表示してください。
また、在庫が10個以下の商品には「⚠ 在庫僅少」と注記してください。
```

## 演習3：ドキュメント連携

### 目的
仕様書とDBスキーマを照合し、未実装・不整合を洗い出す。  
実際の開発現場で発生しがちな「仕様書とコードのずれ」を体験する。

### 使用するMCPツール
- **filesystem**: `workspace/docs/` の仕様書を読み込む
- **sqlite**: DBスキーマ（テーブル定義）を確認する

### 悪いプロンプト例 ❌

```
仕様書とDB確認して
```

**なぜ悪いか**: 何を確認するのか、何を洗い出したいのかが不明。  
「確認しました」で終わってしまい、具体的な問題点が出てこない。

---

### 良いプロンプト例 ✅

```
あなたはシステム監査担当のエンジニアです。

以下の手順でドキュメントとDBの整合性を確認してください。

【手順】
1. workspace/docs/ 配下の仕様書（01〜05）をすべて読み込む
2. SQLiteのDBに接続し、全テーブルのスキーマ（CREATE TABLE文）を取得する
3. 仕様書に記載されているが、DBに実装されていない要素を洗い出す

【チェック観点】
- 仕様書に記載のテーブル・カラムがDBに存在するか
- 仕様書に書かれた制約（NOT NULL / UNIQUE / FK）がDBに設定されているか
- 仕様書内の「TODO」や「未実装」と明記されている箇所

【出力形式】
| 仕様書ファイル | 該当箇所 | 内容 | 重要度（高/中/低） |
の表形式でまとめてください。
```

**なぜ良いか**:
- ①役割：監査担当として客観的視点で確認させる
- ②背景：手順を3ステップで明示
- ③依頼：照合内容を具体的に列挙
- ④制約：チェック観点を3つに絞る
- ⑤出力形式：表形式で重要度付きを指定

---

### チャレンジ課題

```
02_db-design.md の「設計上の注意点」セクションを読み、
実際のDBで懸念が実現していないか（例：stock_countとstock_historyの不整合）を
確認し、現状評価レポートを作成してください。
```

## 演習4：プロンプト改善チャレンジ（チーム対抗）

### 目的
良いプロンプトの5要素を意識して、各自がプロンプトを考え、結果を比較する。  
同じお題でも、プロンプトの書き方によって回答の質がどれほど変わるかを体験する。

### ルール
1. 各自が同じお題に対してプロンプトを書く（5分）
2. 全員が順番に実行して結果を画面共有
3. 「どのプロンプトが最も使いやすい回答を得られたか」をチームで評価

---

### お題1（ログ × DB 横断分析）

**お題**:
> 3月に最も障害が多かった時間帯と、その時間帯に処理されていた注文データを紐付けて、経営会議向けの報告書を作成してください。

**ヒント**:
- 使うデータ: ログファイル（ERROR/WARN）＋ ordersテーブル
- 役割: 「あなたはCTOに報告するSREリーダーです」など
- 出力形式: 経営会議向けなら「箇条書き＋インパクト金額」が有効

**良いプロンプトの例（参考）**:
```
あなたはCTOへ報告するSREリーダーです。

【状況】
3月に複数の障害が発生し、経営会議で報告が必要です。

【依頼】
以下の手順で分析・報告書を作成してください：
1. 3/1〜3/10のログから ERROR・WARN の発生時間帯を集計
2. 最多発生時間帯（1時間単位）を特定
3. その時間帯に orders テーブルで作成・処理された注文を抽出
4. 障害と注文への影響を紐付けて評価

【制約】
- 経営陣向けのため専門用語は避ける
- 金額的インパクト（影響注文の合計額）を必ず含める

【出力形式】
① 障害サマリー（3行以内）
② 最多障害時間帯と発生件数（表）
③ 影響を受けた可能性のある注文一覧
④ 再発防止策の提言（2〜3点）
```

---

### お題2（仕様書 × DB 引き継ぎ資料）

**お題**:
> 新メンバーへの引き継ぎ資料として、このECシステムのデータ構造と注意すべき設計上の問題点をまとめてください。

**ヒント**:
- 使うデータ: 仕様書全ファイル＋DBスキーマ
- 役割: 「あなたはこのシステムを3年間担当してきたバックエンドリードです」
- 出力形式: 新メンバーが読むなら「概要 → 詳細 → 注意点」の構成が読みやすい

**良いプロンプトの例（参考）**:
```
あなたはこのECシステムを3年間担当してきたバックエンドリードです。

【状況】
来月から新メンバーが参画します。
システムの全体像とDB設計の注意点を引き継ぐ必要があります。

【依頼】
workspace/docs/ の仕様書を全て読み込み、DBスキーマも確認した上で、
以下の内容を含む引き継ぎ資料を作成してください。

【含める内容】
1. システムの全体像（5分で理解できるレベル）
2. 主要テーブルの関係と業務上の役割
3. 「ここだけは注意して」という設計上のクセ・落とし穴
4. 既知のTODO・技術的負債のリスト

【制約】
- 入社1年目のエンジニアでも理解できる言葉で書く
- 専門用語を使う場合は括弧内で簡単な説明を付ける

【出力形式】
Markdown形式で、見出しを使って構造化してください。
```

---

### 評価基準（チームで議論してみよう）

| 観点 | チェックポイント |
|------|----------------|
| 網羅性 | 重要な情報が抜けていないか |
| 正確性 | DBの実データ・ログの内容と矛盾していないか |
| 読みやすさ | 対象読者（経営陣/新人など）に適した表現か |
| アクション可能性 | 読んだ人が次に何をすべきかわかるか |

---

## まとめ：プロンプト改善のコツ

1. **まず悪いプロンプトで試す** → どれだけ曖昧な回答が返るか体感する
2. **5要素を一つずつ追加** → どの要素が回答改善に最も効いたかを観察する
3. **出力形式の指定は強力** → 「表形式で」「箇条書きで」「3行以内で」を試す
4. **役割指定は文脈を作る** → 「SREエンジニア」「経営企画担当」で視点が変わる
5. **制約で絞り込む** → 情報が多すぎる場合は「〜は除く」「〜のみ対象」を使う
